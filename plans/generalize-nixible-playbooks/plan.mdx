<RichText id="intro">

# Generalize nixible playbooks into a declarative home-manager option

Today the nixible integration is one bespoke file, `modules/nixible/claude-code.nix`, that hand-wires three things specifically for Claude Code: the playbook data, a `nix run .#nixible:claude-code` app, and a `den.aspects.claude-code` aspect with a `home.activation.installClaudeCode` hook. Adding a second playbook means copy-pasting all of that boilerplate and inventing new hardcoded names.

This plan replaces the per-playbook boilerplate with a single declarative option: **`nixible.playbooks.<name>`**, a home-manager option that any aspect can set. One small infra module declares the option and merges every declared playbook into a **single nixible run** on activation. Because playbooks live in home-manager config, whether a playbook is applied is governed entirely by **whether the aspect carrying it is included** in the active configuration — exactly like `programs.claude-code.skills.*` works today. The aspect name has nothing to do with whether it carries a playbook.

After this, adding a playbook is: write the Ansible data structure, assign it to `nixible.playbooks.<name>` under whichever aspect should own it (`justin` for global, `amperity`/`work` for work-only, a host aspect for host-only), done. No new aspect, no new activation-hook name, no flake-level wiring.

</RichText>

<RichText id="reuse-heading">

### What this reuses vs. adds

</RichText>

<Callout id="reuse-callout" tone="info">

**Reuses** the exact composition model already in the repo: aspects merge their `homeManager` modules into one home config, and option assignments across aspects merge (proven by `amperity/claude-code.nix` contributing `programs.claude-code.skills.*` to the work-only `amperity` aspect). It also reuses nixible's own `mkNixibleCli` builder and the `lib.hm.dag.entryAfter ["writeBoundary"]` activation pattern from the current code.

**Adds** one infra module that *declares* a `nixible.playbooks` option and *merges* all enabled entries into a single nixible call behind one activation hook. That's the only genuinely new piece — everything else is moving existing data behind the option.

</Callout>

## The option schema

`nixible.playbooks.<name>` is an attrset of submodules. Each entry carries just the inputs `mkNixibleCli` needs, plus an `enable` toggle. All enabled entries are merged into one nixible config and run in a single `ansible-playbook` call (nixible's `playbook` field is itself a list of plays), so adding playbooks costs no extra process startups.

</RichText>

## Implementation

### 1. New infra module: declare the option, merge into one run

This is the only new file. It is imported normally by `import-tree ./modules` (it has no `/_` in its path, so it is not filtered out). It defines an always-on aspect `den.aspects.nixible` whose home-manager module declares `nixible.playbooks` and folds every *enabled* entry into a single combined nixible config — concatenating the plays, unioning the dependencies, and merging the collections — fired by one `home.activation.nixible` hook. The `inputs` arg is captured from the file's top-level scope, exactly as the current `claude-code.nix` does.

</RichText>

<AnnotatedCode
  id="infra-module"
  filename="modules/nixible/playbooks.nix"
  language="nix"
  code={"{inputs, ...}: {\n  # Always-on aspect: declares `nixible.playbooks` and runs every declared\n  # playbook. All enabled playbooks are merged into a SINGLE nixible CLI and\n  # run in one `ansible-playbook` call. Plays run in alphabetical order of key.\n  den.aspects.nixible.homeManager = {\n    config,\n    lib,\n    pkgs,\n    ...\n  }: let\n    inherit (lib) types mkOption attrValues filterAttrs concatMap foldl' unique optionalAttrs hm;\n    cfg = config.nixible.playbooks;\n    enabled = attrValues (filterAttrs (_: pb: pb.enable) cfg);\n\n    # Fold every enabled playbook into a single nixible config.\n    combined = {\n      dependencies = unique (concatMap (pb: pb.dependencies) enabled);\n      collections = foldl' (acc: pb: acc // pb.collections) {} enabled;\n      playbook = concatMap (pb: pb.playbook) enabled;\n    };\n\n    cli = (inputs.nixible.lib {inherit pkgs lib;}).mkNixibleCli combined;\n\n    playbookType = types.submodule {\n      options = {\n        enable = mkOption {\n          type = types.bool;\n          default = true;\n          description = \"Include this playbook in the activation run.\";\n        };\n        dependencies = mkOption {\n          type = types.listOf types.package;\n          default = [];\n          description = \"Packages on PATH while the playbooks run.\";\n        };\n        collections = mkOption {\n          type = types.attrsOf (types.attrsOf types.str);\n          default = {};\n          description = \"Ansible Galaxy collections (version + hash).\";\n        };\n        playbook = mkOption {\n          type = types.listOf types.attrs;\n          description = \"One or more Ansible plays, as a Nix data structure.\";\n        };\n      };\n    };\n  in {\n    options.nixible.playbooks = mkOption {\n      type = types.attrsOf playbookType;\n      default = {};\n      description = \"Nixible playbooks applied during home-manager activation.\";\n    };\n\n    config.home.activation = optionalAttrs (enabled != []) {\n      nixible = hm.dag.entryAfter [\"writeBoundary\"] ''\n        $DRY_RUN_CMD ${cli}/bin/nixible\n      '';\n    };\n  };\n}\n"}

<RichText id="example-heading">

## Adding a new playbook after this

A new playbook is a single file (or a few lines in an existing aspect file). Example: an `atuin` shell-history sync, scoped global by attaching to `justin`. Drop the file under `modules/`; `import-tree` picks it up; it applies on the next activation. To make it work-only instead, the exact same block goes under `den.aspects.amperity.homeManager`.

</RichText>

<RichText id="tradeoff-heading">

## Key decision: the `nix run` apps go away

Today `perSystem.nixible.claude-code` exposes `nix run .#nixible:claude-code` as a side effect. Once playbooks live inside per-aspect home-manager config (per user, per host), there is no single flake-level list to generate those apps from. **Decision: drop the standalone apps entirely** — the activation hook is the real integration and the apps were never the point. No manual-run path and no `home.packages` exposure; re-running a playbook means re-activating home-manager. This also removes the now-unused `inputs.nixible.flakeModule` import repo-wide.

</RichText>
